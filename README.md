# Project Report: Multi-Switch Flow Table Analyzer

## 1. Project Description
The **Multi-Switch Flow Table Analyzer** is a Software-Defined Networking (SDN) application designed to provide centralized visibility and management across a distributed network. Utilizing the **Ryu Controller** and **OpenFlow 1.3** protocol, the project transforms traditional network switches into programmable entities managed by a centralized "brain."

The core objective of this system is to bridge the gap between network forwarding and network monitoring. While the controller manages how data moves between hosts, it simultaneously audits the switches' internal flow tables. By dynamically polling these switches, the application identifies which forwarding rules are **ACTIVE** (processing live traffic) and which are **IDLE**, enabling network administrators to optimize performance and detect unused or redundant security policies.

## 2. Technical Methodology (What I Did)

### Phase 1: Infrastructure Optimization
I initially attempted to build the SDN environment using Docker and WSL2. However, I diagnosed a **FATAL kernel-level error** where the `openvswitch` module was unavailable in the containerized environment. 
* **Action:** I migrated the entire development stack to a standalone **Ubuntu 22.04 EC2 instance**.
* **Result:** This provided the necessary kernel support for Open vSwitch 3.x, ensuring a stable environment for the SDN data plane.

### Phase 2: Custom Topology Engineering
I developed a custom Python script (`topo.py`) to define the network structure in Mininet. 
* **Design:** 2 Open vSwitch instances ($s1, s2$) and 3 hosts ($h1, h2, h3$).
* **Linkage:** $h1$ and $h2$ were connected to $s1$, while $h3$ was connected to $s2$, with a cross-switch link connecting $s1$ and $s2$.
* **Protocol:** Configured to use **OpenFlow 1.3** for modern matching capabilities.

### Phase 3: Controller Logic & Application Development
I authored a custom Ryu application (`analyzer.py`) implementing a dual-functional logic:
1. **Reactive MAC Learning:** The controller handles unknown traffic by learning MAC addresses and their corresponding ports, then pushes flow entries to the switches to enable end-to-end connectivity.
2. **Asynchronous Monitoring:** I implemented a background thread using `ryu.lib.hub` that triggers `OFPFlowStatsRequest` messages to all connected switches every few seconds.

### Phase 4: Dynamic Flow Analysis
I created a processing engine to interpret the statistics returned by the switches. 
* **Logic:** The system extracts the `packet_count` for each rule.
* **Classification:** It classifies rules as **ACTIVE** if the count is increasing and **IDLE** if it remains static.
* **Verification:** I validated the controller logs against hardware-level outputs using `sudo ovs-ofctl dump-flows`.

### Phase 5: Verification with Wireshark (Tshark)
To complete the "Multi-Switch" analysis, I utilized **Tshark** to capture control-plane traffic on port 6653. This provided visibility into the OpenFlow handshake, confirming that the `FLOW_MOD` messages were correctly generated by my analyzer and accepted by the switches.

## 3. Key Achievements
- **0% Packet Loss:** Verified connectivity across multiple switches via `pingall`.
- **Centralized Visibility:** Real-time console logs showing the status of flow entries for all switches in the network.
- **Protocol Compliance:** Fully operational implementation of the OpenFlow 1.3 standard.
